This module blends programming logic with foundational maths, and the past papers (Spring, Summer, Autumn and Winter 2025, plus Spring 2026) consistently reward students who can trace through an algorithm step-by-step rather than just describe what it "generally does." This guide focuses on the areas that repeat most.

1. Pseudocode and flowchart conventions examiners expect

Marks are frequently lost not because the logic is wrong, but because the pseudocode or flowchart doesn't follow the conventions the mark scheme expects. Stick to these habits:

  • Use clear, consistent keywords: IF...THEN...ELSE...ENDIF, WHILE...ENDWHILE, FOR...NEXT
  • In flowcharts: ovals for start/end, rectangles for process steps, diamonds for decisions, parallelograms for input/output
  • Always show variable initialisation before it's used (e.g. total = 0 before a loop that adds to total)

2. Sequence, selection and iteration

These are the three building blocks of every algorithm, and questions often ask you to identify or write examples of each:

  • Sequence: instructions executed one after another, in order
  • Selection: a decision point where the program branches based on a condition (IF/ELSE)
  • Iteration: a loop that repeats a set of instructions, either a fixed number of times (FOR) or until a condition is met (WHILE)

A common exam task is to trace a piece of pseudocode using a trace table, tracking how each variable changes on every iteration:

count = 0
total = 0
WHILE count < 5
    total = total + count
    count = count + 1
ENDWHILE

Exam tip: When asked to write a compound condition (e.g. "a discount applies if the customer is a member AND has spent over $50"), write it exactly the way it would appear in code: IF isMember = TRUE AND totalSpent > 50 THEN — examiners check both the logic and correct use of AND/OR.

4. Number systems — binary, denary and hexadecimal

Conversion questions are near-guaranteed. The methods to know cold:

  • Binary to denary: multiply each bit by its place value (128, 64, 32, 16, 8, 4, 2, 1 for an 8-bit number) and sum the results where the bit is 1. E.g. 10110100 = 128+32+16+4 = 180.
  • Denary to binary: repeatedly divide by 2, recording the remainders, then read them bottom to top.
  • Binary to hexadecimal: split the binary number into groups of 4 bits (nibbles) starting from the right, then convert each nibble to its hex digit (0-9, A-F).

Practising 4-5 conversions of each type by hand until they're automatic is worth more than reading the theory again.

5. Algorithm efficiency in plain language

At Level 4 you're not expected to formally prove Big O notation, but you are expected to explain, in plain terms, why one approach to a problem is more efficient than another — for example, why a binary search is faster than a linear search on a sorted list, because it eliminates half the remaining data with each comparison instead of checking every item one by one.

6. Common mistakes that cost marks

  • Inconsistent pseudocode keywords — mixing IF/ENDIF with different bracket styles confuses markers and can lose clarity marks.
  • Skipping steps in trace tables — show every iteration, even if the pattern seems obvious.
  • Sign errors in binary-denary conversion — double-check place values, especially with 8-bit vs 16-bit questions.
  • Vague justification for algorithm choice — "it's faster" isn't enough; explain why in terms of how many comparisons or steps are needed.
