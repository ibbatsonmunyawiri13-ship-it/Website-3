The Databases module rewards students who can actually build a small database on paper — draw the diagram, identify the keys, normalise it, and write a query against it — rather than students who've only memorised definitions. This guide works through each stage in the order examiners usually test it.

1. Why databases exist — the exam-safe explanation

If a question asks you to explain the advantages of a database system over a flat file (spreadsheet-style) system, examiners are looking for specific, named advantages rather than a vague "it's more organised." The safe list to know:

  • Reduced data redundancy — data is stored once and referenced, rather than repeated across multiple files
  • Improved data integrity — validation rules and constraints prevent inconsistent or invalid data
  • Better data security — access can be restricted at the table, row, or field level for different users
  • Concurrent access — multiple users can safely read and write data at the same time

2. Primary keys, foreign keys and referential integrity

This is one of the most consistently tested areas. Be precise with your definitions:

  • Primary key: a field (or combination of fields) that uniquely identifies every record in a table. It cannot be null and cannot repeat.
  • Foreign key: a field in one table that references the primary key of another table, used to create a relationship between the two.
  • Referential integrity: the rule that a foreign key value must either match an existing primary key value in the related table, or be null — this prevents "orphaned" records that reference something that doesn't exist.
Common exam phrasing: "Explain why referential integrity is important in a relational database." Safe answer: it prevents inconsistent data, such as an order record referencing a customer ID that has been deleted, which would otherwise leave the database in an invalid state.

3. Entity-Relationship diagrams and relationship types

You'll be expected to read or draw a simple ER diagram and correctly identify the relationship type between two entities:

RelationshipExampleNotation
One-to-one (1:1)One employee has one staff ID card1 — 1
One-to-many (1:M)One customer places many orders1 — M
Many-to-many (M:N)Students enrol in many courses; courses have many studentsM — N

Many-to-many relationships are usually the trickiest part of an exam question, because they require a linking (junction) table in the actual database design — you can't implement M:N directly in a relational table structure, so this is a frequent follow-up question.

4. Normalisation — 1NF, 2NF, 3NF without the confusion

Normalisation questions are marked on process, not just the final answer, so show your working. Think of each stage as removing one specific type of problem:

  1. 1NF (First Normal Form): remove repeating groups — each field should hold a single, atomic value (no lists of values crammed into one field), and there should be a primary key.
  2. 2NF (Second Normal Form): remove partial dependency — every non-key field must depend on the whole primary key, not just part of it (only relevant when you have a composite key).
  3. 3NF (Third Normal Form): remove transitive dependency — no non-key field should depend on another non-key field; every non-key field should depend directly on the primary key only.
Worked mini-example: A table storing OrderID, ProductID, ProductName, and ProductPrice violates 3NF, because ProductName and ProductPrice depend on ProductID, not directly on OrderID. The fix: split ProductName and ProductPrice into a separate Products table, linked by ProductID as a foreign key.

5. SQL essentials examiners expect you to write

You should be comfortable writing, not just reading, these query types:

  • SELECT column1, column2 FROM table WHERE condition; — basic filtering
  • SELECT * FROM orders JOIN customers ON orders.customerID = customers.customerID; — joining two related tables
  • INSERT INTO table (col1, col2) VALUES (val1, val2); — adding a new record
  • UPDATE table SET column = value WHERE condition; — modifying existing records
  • DELETE FROM table WHERE condition; — removing records

A frequent exam trap is forgetting the WHERE clause on UPDATE or DELETE statements — always mention in your written answer that omitting it would affect every row in the table, since examiners often award a mark specifically for noting this risk.

6. Common mistakes that cost marks

  • Mixing up primary and foreign keys in diagram-reading questions — always check which table "owns" the unique ID versus which table is referencing it.
  • Stopping normalisation at 1NF when the question asks for 3NF — show every stage, not just the final table structure.
  • Writing SQL with syntax that doesn't match what was taught — stick to the standard syntax patterns shown in your course materials rather than guessing.
  • Vague answers on "why normalise" — the precise reasons are reducing redundancy and preventing update/insert/delete anomalies, not just "to make it neater."
